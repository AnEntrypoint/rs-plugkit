## Resolved mutable: vecmig-recency-dedup-semantics

Decision: given shared-db-inmemory-not-persisted blocks any read-path cutover today (rssearch_vectors would be wiped every watcher restart, unlike the disk-persisted flat-JSON host_vec_search), the recency-decay/COS_FLOOR/Jaccard-dedup semantics do not need replication yet because the read path is not switching. If/when project_db_path is wired to a real file and a future session flips the read path, the correct replication is: SQL vector_top_k gives cosine distance only (no recency/floor/dedup), so those three steps must run in Rust after fetching a k*5-wide candidate pool from vector_top_k, mirroring code_index.rs::search()'s existing pool-then-filter pattern (pool = k.saturating_mul(5).max(20)) rather than being pushed into SQL. Deferred, not solved, but the shape of the eventual fix is now on record so it is not re-discovered from scratch.
