Freshness-marker anti-pattern and fix, generalized: a bare-existence marker file ("has this run completed") that gates re-running expensive/batched work is unsafe whenever the underlying work can be partial (batch cap, deferred items) or the source data can grow after the marker was written -- the marker silently strands all future growth forever with no error surfaced, because existence alone carries no information about what state it was written against. The fix pattern (mirrors rs-plugkit's codeinsight sync-before-emit stored_digest/current_digest convergence check): compute a deterministic digest+count over the current source state, embed it INSIDE the marker body (not just the marker's existence), and gate re-run on digest-match rather than mere presence. A marker whose body fails to parse into the expected shape (e.g. a legacy bare-string marker predating the fix) must be treated as absent/stale, which gives automatic backward-compatible migration for free with zero separate migration code path. Applies to any "done" flag file across the codebase, not just memory/export mechanisms.
